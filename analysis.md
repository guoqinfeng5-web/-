
## 1. 前期调研

### 1.1 需求拆解

用户问题大致分为三类：

1. **推荐/解释类**（“这本书适合谁、给学习路径、推荐高分书”）  
   - 优点：大模型擅长组织语言与给建议  
   - 风险：容易“凭空推荐馆内不存在的书”  

2. **事实细节类**（“作者/评分/库存/借阅次数/分类数量”）  
   - 需要从数据库或可验证数据源取数  
   - 风险：若只靠 RAG 片段，字段可能不全；若只靠模型，容易乱编  

3. **统计分析类**（“TopN/分布/占比/按分类汇总/我的借阅分析”）  
   - 最适合直接走数据库聚合查询（SQL）  
   - 风险：SQL 注入与误写、口径不统一、返回空时总结容易过度推断  

### 1.2 方案对比

- **只做纯 LLM 问答**：实现最快，但幻觉风险最高（不适合“馆藏/借阅”这类必须真实的数据域）
- **只做 RAG**：能减少“编造”，但对统计/聚合场景仍不稳定（片段不足/口径不准）
- **RAG + NL2SQL（推荐组合）**：  
  - 推荐/解释类：RAG 提供证据 + LLM 组织语言  
  - 统计/聚合类：NL2SQL 直接查库 + LLM 总结  
  - 两条链路互补，且可以通过 `/查询` 强制走可追溯分析链路

结论：在两天内，选择 **RAG + NL2SQL + 短期记忆** 三模块的组合，能在可控复杂度下覆盖主要场景。

## 2. 总体设计思路（三个核心模块）

本 Demo 的 AI 能力按三模块拆分（彼此解耦、便于扩展/替换）：

1) **RAG 馆藏问答模块**（“以馆藏为依据的自然语言回答”）  
2) **NL2SQL 统计分析模块**（“以数据库查询结果为依据的可追溯分析”）  
3) **短期记忆模块**（“保留上下文但控制长度”）  

```mermaid

Nl2sql --> ChatModel
Memory --> Router
```

## 3. 模块一：RAG（向量嵌入 + 检索 + 上下文注入）

### 3.1 设计目标

- 让模型回答“书籍相关问题”时尽量基于馆藏证据，减少幻觉
- 两天内可交付：优先选“可跑通、可解释、可调试”的实现

### 3.2 关键实现点（原理）

1. **向量化入库（Embedding / ingest）**  
   - 将每本书拼成结构化文本 `featureText`（字段包含：`title/author/category/price/score/stock/tags/borrowCount/summary`）  
   - 调用 Embedding 模型生成向量  
   - 以内存结构保存 `TextSegment + Embedding`  

2. **向量检索（search）**  
   - 对用户问题向量化  
   - 逐条计算余弦相似度（cosine similarity）  
   - 排序取 TopK 片段作为上下文（例如 K=5）  

3. **上下文注入（prompt 拼接）**  
   - 将 TopK 片段拼到 `[RAG 馆藏背景]`  
   - 由 ChatModel 在该上下文基础上生成答案  

### 3.3 取舍与后续扩展

- **当前取舍**：向量库使用内存 List（实现快、无需额外部署）  
  - 优点：调试简单  
  - 缺点：重启需重新 ingest；数据量大时检索 O(N)  
- **可扩展方向**：接入向量数据库（Milvus/pgvector/ES 向量检索），支持增量更新与更快检索

## 4. 模块二：NL2SQL（Tool 只读查库 + 安全校验 + 数据驱动总结）

### 4.1 设计目标

- 面向“统计/聚合/排行/分布/我的借阅”等问题，提供**可追溯**结论
- 让“数据口径”由 SQL 保证，而非模型自由发挥

### 4.2 关键实现点（原理）

1. **触发策略**  
   - 用户输入以 `/查询` 或 `/query` 开头：强制走 NL2SQL  
   - 或命中统计意图：自动走 NL2SQL  

2. **Tool 调用查库**  
   - 模型生成 SQL，通过 Tool `queryLibraryDatabase(sql)` 执行  
   - Tool 返回 JSON 数组（每行一对象）  

3. **SQL 安全校验**  
   - 仅允许单条 `SELECT`  
   - 白名单表：`books`、`borrow_records`  
   - 必须 `LIMIT` 且 ≤100  
   - 禁止多语句与危险关键字（INSERT/UPDATE/DELETE/DROP 等）  

4. **口径约束（避免“总结过度推断”）**  
   - 未归还：`return_time IS NULL`；已归还：`IS NOT NULL`  
   - 若只查了未归还结果为空，只能说“当前无未归还/在借”，禁止推断“所有都已归还”（需要额外总数对比才能下结论）

### 4.3 取舍与后续扩展

- **当前取舍**：由大模型生成 SQL + 严格校验后执行  
  - 优点：覆盖面广、迭代快  
  - 风险：模型生成的 SQL 可能不理想，需要用 schema/prompt 约束与校验兜底  
- **可扩展方向**：  
  - 加入“SQL 模板库”或“查询意图到 SQL 的半结构化映射”，减少模型自由度  
  - 加入可观测性：记录 normalized SQL、rows count（不输出敏感信息）

## 5. 模块三：短期记忆（Redis 历史 + 摘要压缩）

### 5.1 设计目标

- 让系统能记住用户偏好与上下文（例如“偏好 Java/数据库方向”）
- 避免对话无限增长导致 prompt 过长、成本升高、模型不稳定

### 5.2 关键实现点（原理）

1. **历史存储**：Redis List  
   - key：`ai:chat:history:v2:<sessionId>`  
   - 保存每轮 `user` 与 `assistant` 消息  

2. **压缩策略**：超过阈值后摘要  
   - 当历史条数超过 `MAX_HISTORY_MESSAGES`  
   - 将“前半段历史”摘要成一条 `SUMMARY:` system 消息  
   - trim 掉已摘要部分，保留摘要 + 最近对话  

### 5.3 取舍与后续扩展

- **当前取舍**：摘要压缩采用“前半段摘要 + 保留后半段”  
  - 优点：实现简单、可控  
  - 缺点：摘要质量依赖模型；可通过更结构化的摘要提示词提升稳定性  
- **可扩展方向**：引入长期记忆（用户画像/偏好表）、或在检索时将摘要也纳入检索条件

## 6. 技术选型依据

### 6.1 后端与数据

- **Spring Boot + MyBatis-Plus + MySQL**：快速搭建 CRUD 与分页/排行查询；面试官易读易理解
- **Redis**：适合做对话历史与滑动窗口摘要的存储（列表操作与 trim 方便）

### 6.2 AI 框架与模型

- **LangChain4j**：\n  - 以 Java 生态为主，适配 Spring Boot\n  - AiServices + Tool 调用模式适合 NL2SQL\n  - 便于把“查库工具”绑定给模型并自动完成“生成SQL→执行→总结”循环\n+- **Embedding（text-embedding-v4）**：用于向量检索（RAG）\n+- **ChatModel（Qwen/Qwen3.5-35B-A3B）**：用于自然语言总结与回答组织

### 6.3 前端

- **Vue 3 + Vite + Element Plus**：两天内可快速出可交互 UI，开发体验好，组件齐全

## 7. 当前能力边界

- “实时天气”等问题不在馆藏数据域内：若不接外部 API，模型只能给通用建议/追问城市，无法保证真实实时天气
- RAG 只提供“检索证据”，统计口径仍建议走 `/查询` 的 NL2SQL 链路
